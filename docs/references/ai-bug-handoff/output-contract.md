# AI Bug 交接产物约定

本文定义自动化执行后，面向 AI 修复代理、开发同学和测试同学的 Bug 交接输出约定。目标是让接手方在不重新跑全流程的情况下，也能快速理解失败现场、影响范围和修复边界。

## 1. 输出目标

需要同时覆盖下面三类需求：

- 让人能看懂：失败发生在哪个用例、哪一步、预期是什么、实际发生了什么。
- 让 AI 能消费：字段稳定、命名稳定、上下文完整，适合继续生成修复方案或补丁。
- 让流程能追踪：同一轮执行的结果、附件和 Bug 列表可以相互关联。

## 2. 最低输出清单

一次正式失败交接，至少应包含以下内容：

1. `case_results.xlsx`
   记录本轮执行的用例级结果、统计字段和关键失败摘要。
2. `bug_list.xlsx`
   记录需要继续跟进的失败项，每条记录对应一个待处理 Bug。
3. Bug Markdown 任务单
   面向 AI 或开发的单 Bug 说明文档，建议按模板生成。
4. 失败附件目录
   包含截图、HTML、DOM 片段、网络日志、控制台日志或其他辅助证据。

## 3. `bug_list.xlsx` 字段要求

每条 Bug 至少要包含以下信息：

1. 基本标识
   包括 `bug_id`、`title`、`run_id`、`case_id`、`module`、`priority`。
2. 失败定位
   包括 `failed_step_no`、`failed_step_name`、`page_name`、`target`。
3. 结果对比
   包括 `expected_result`、`actual_result`、`failure_message`。
4. 证据索引
   至少能反查截图、HTML、日志或 DOM 快照所在路径。
5. 修复边界
   包括 `change_scope`、`do_not_change`，用于约束后续修改范围。
6. 状态字段
   包括当前是否待处理、是否已交接、是否需要人工复核等状态信息。

## 4. `case_results.xlsx` 字段要求

用例结果表重点服务于测试汇总和阶段报告，建议包含：

- 用例基础信息
  包括用例编号、名称、模块、标签、负责人、执行环境。
- 执行结果
  包括 `passed`、`failed`、`skipped`、`blocked` 等状态。
- 失败摘要
  用一句中文说明失败的核心现象，方便快速筛选。
- Bug 关联
  如果该用例已生成 Bug，需要能回链到 `bug_id` 或 Bug 任务单。
- 统计字段
  至少包含总数、通过数、失败数、通过率等聚合信息。

## 5. Bug Markdown 任务单要求

单个 Bug 的 Markdown 任务单应尽量做到“拿到就能接着修”：

- 标题直接描述业务影响，不只写断言失败。
- 明确失败步骤、页面对象、关键目标元素或关键接口。
- 同时写清预期结果和实际结果，避免只有报错堆栈。
- 附件路径要可追溯，最好一眼能定位截图和 HTML。
- 如果已有怀疑方向，要写在“可能原因”或“排查建议”中，而不是散落在备注里。

## 6. 附件与证据链要求

失败证据建议遵循“先结论、后现场、再上下文”的组织方式：

- 先给失败截图，方便快速感知页面状态。
- 再给 DOM / HTML 片段，帮助确认元素结构是否变化。
- 再给控制台日志、网络日志，帮助判断是否有前端或接口异常。
- 如果是接口断言失败，最好补充请求方法、路径、状态码和关键响应摘要。
- 如果是定位器失效，最好补充当前命中的页面对象 key 和备用定位线索。
- Manifest 或元数据里应保留附件相对路径，方便 AI 和脚本自动检索。

## 7. 命名与稳定性要求

为了让 AI 和后续脚本更稳定地消费这些产物，建议遵守以下规则：

- 字段名稳定，不随临时习惯改写。
- 同一含义尽量只保留一种主字段名。
- 中文说明可以增强，但不要替换英文主键。
- 路径最好使用相对工作目录可解析的形式。

## 8. 交接前自检清单

输出 Bug 交接包之前，至少确认以下事项：

1. 失败用例都能在 `case_results.xlsx` 中找到。
2. 每条待处理 Bug 都能在 `bug_list.xlsx` 中找到。
3. Bug 任务单里的截图、HTML、日志路径都存在。
4. 预期结果和实际结果不是空白或互相重复。
5. 修改范围和禁止修改范围已经写清楚。
6. 统计信息与实际用例结果一致。
