# 项目协作与编码约束

本文档用于约束本项目后续的代码编写、目录使用、注释风格、日志输出、缺陷建模和文档规范。  
后续所有参与本项目的开发者或智能体，在没有更高优先级指令的前提下，都应遵守本文件。

## 1. 总体原则

1. 本项目优先保证可维护性、可追踪性和可扩展性，不追求一次性脚本式实现。
2. `Excel`、对象库、执行引擎、结果模型、报告生成、缺陷分析必须分层，不允许混写。
3. 所有报告、缺陷文档和结果产物都必须来自统一结果模型，不允许各模块自行拼装独立结果格式。
4. 非必要文档统一使用中文；字段名、代码标识、文件名、路径、协议名保持英文。
5. 在没有明确技术选型前，不要过早引入与语言或框架强绑定的结构。

## 2. 目录使用约束

项目目录职责固定如下：

- `configs/`
  存放环境配置、运行配置、全局配置模板。
- `excels/`
  存放测试用例工作簿、测试数据模板。
- `object_repo/`
  存放对象库工作簿或对象映射配置。
- `outputs/`
  存放每轮执行结果，不允许手工修改历史结果。
- `src/core/`
  存放主流程编排、应用入口、全局上下文。
- `src/parser/`
  仅负责输入解析，不负责执行逻辑。
- `src/validator/`
  仅负责校验，不直接访问浏览器驱动。
- `src/runner/`
  负责执行用例和步骤，不负责产物渲染。
- `src/actions/`
  存放动作关键字执行器。
- `src/asserts/`
  存放断言关键字执行器。
- `src/objects/`
  存放对象库解析、定位策略、页面对象辅助逻辑。
- `src/waits/`
  存放等待、重试、同步控制逻辑。
- `src/artifacts/`
  存放证据采集逻辑。
- `src/results/`
  存放统一结果模型构建逻辑。
- `src/bugs/`
  存放失败归一化、去重、缺陷聚类逻辑。
- `src/reports/`
  存放测试报告、结果表、缺陷文档生成逻辑。
- `src/utils/`
  存放通用工具，但不得堆积业务逻辑。
- `templates/`
  存放文档模板、报告模板、导出模板。
- `logs/`
  存放运行日志或调试日志。
- `tests/`
  存放单元测试、集成测试、样例数据。

## 3. 分层边界约束

1. `parser` 只做解析，不做业务判断。
2. `validator` 只输出校验结果，不直接修改原始输入数据。
3. `runner` 不直接生成最终报告文件。
4. `reports` 不直接读取 `Excel`，只读取统一结果模型。
5. `bugs` 模块不依赖前端展示层，只依赖结果模型和归一化规则。
6. `objects` 模块不感知用例筛选逻辑，只处理对象解析和定位。
7. `artifacts` 模块只负责采集与索引，不负责缺陷判断。

## 4. 输入模型约束

1. `Case_Index` 一行只表示一条测试用例。
2. `Case_Steps` 一行只表示一个步骤。
3. `Case_Steps` 的 `step_type` 只能是 `action` 或 `assert`。
4. 动作步骤只填写 `action_key`，断言步骤只填写 `assert_key`。
5. `target` 必须优先写对象别名，不允许在普通用例中直接写原始 XPath。
6. `data_set_id` 必须关联 `Test_Data`，不允许在步骤中大量硬编码业务数据。
7. 所有枚举值必须来自 `Dictionary` 或代码中冻结的枚举定义。

## 5. 对象库约束

1. 对象库是 `Excel` 用例与实际定位器之间的唯一映射层。
2. 每个对象必须有稳定的 `object_key`。
3. 每个关键对象至少要有一个主定位器，建议有一个备用定位器。
4. 对象必须归属于明确页面，不允许大量“全局无归属对象”。
5. 对象命名应体现业务语义，不体现定位实现细节。
6. 优先使用稳定定位方式，例如 `testid`、`id`、`name`，谨慎使用脆弱 XPath。
7. 对关键对象应补充 `ai_component_hint`、`frontend_file_hint`、`api_hint`。

## 6. 统一结果模型约束

1. 所有输出产物必须基于统一结果模型生成。
2. 统一结果模型至少包括：
   - `run_info`
   - `execution_summary`
   - `case_results`
   - `step_results`
   - `artifacts`
   - `bugs`
   - `bug_case_links`
   - `generation_outputs`
3. 不允许测试报告、`bug Excel`、`AI bug` 文档各自维护独立字段体系。
4. 字段新增时，优先修改统一结果模型，再同步更新产物生成器。

## 7. 缺陷分析约束

1. 不是所有失败都自动生成产品缺陷。
2. 自动化脚本错误、环境错误、测试数据错误必须与产品缺陷分开管理。
3. 缺陷归并必须基于归一化和 `dedup_key` 模板，不允许简单按错误文本直接去重。
4. 必须保留 `dedup_key`、`root_signature` 和归并依据，便于解释和追溯。
5. 一个正式 `bug` 可以关联多条失败用例，但主用例应优先选择证据最完整的记录。

## 8. 代码编写约束

1. 代码标识使用英文，避免拼音和无语义缩写。
2. 变量、函数、类、模块命名要体现职责，不允许使用 `data1`、`tmp`、`do_it` 这类模糊命名。
3. 单个模块应保持职责单一，不要在一个文件中同时处理解析、执行、渲染三类逻辑。
4. 公共逻辑优先抽象成明确的函数或类，不允许把大段重复逻辑散落在多个模块。
5. 写实现前先确认归属模块，避免“先写能跑再挪位置”导致结构失控。
6. 未确定为通用能力前，不要过度抽象。
7. 涉及关键业务判断的地方，优先可读性，避免炫技式写法。

## 9. 注释约束

1. 注释只解释“为什么这样做”或“这段逻辑的边界条件”，不解释显而易见的代码行为。
2. 注释应简洁，避免整段重复翻译代码。
3. 文档类说明优先中文。
4. 代码内注释优先简洁中文；若文件已有明确英文风格或受工具链约束，可使用英文。
5. 以下场景建议加注释：
   - 失败归一化规则
   - `dedup_key` 生成逻辑
   - 对象定位兜底策略
   - 特殊等待或重试原因
   - 与外部格式约定强相关的地方
6. 以下场景不建议加注释：
   - 简单赋值
   - 明显的条件判断
   - 单行直接调用

## 10. 日志与错误处理约束

1. 日志应结构化，至少包含：
   - `run_id`
   - `case_id`
   - `step_no`
   - `module`
   - `status`
   - `message`
2. 日志文本应可读，避免只输出原始异常堆栈而没有上下文。
3. 敏感数据必须脱敏后再写入日志。
4. 解析错误、校验错误、执行错误、环境错误、产品缺陷应明确分类。
5. 用户可修复的输入问题，应输出清晰错误信息和定位信息。
6. 系统异常应保留原始异常信息，但对外输出应附加业务上下文。

## 11. 配置与常量约束

1. 环境地址、浏览器选项、超时配置、重试策略不得写死在业务执行逻辑里。
2. 枚举、状态值、失败分类、定位类型应集中定义。
3. 模板字符串、报告标题、导出表头等应集中管理，不要散落在多个文件中。

## 12. 测试与验证约束

1. 解析器必须有样例输入测试。
2. 校验器必须覆盖必填、唯一性、跨表关联、枚举值校验。
3. `dedup_key` 生成逻辑必须有覆盖不同失败类型的测试。
4. 报告和缺陷产物生成器至少要有样例结果快照验证。
5. 修复缺陷时，优先补对应层级测试，而不是只改主流程代码。

## 13. 文档约束

1. 非必要文档统一使用中文。
2. 文档应优先写清楚边界、输入输出、字段定义、异常场景。
3. 新增模块时，若职责不明显，应补充简短说明文档或模块头注释。
4. 若修改了 `Excel` 结构、对象库结构、结果模型、`dedup_key` 规则，必须同步更新相关文档。

## 14. 提交前自检约束

每次提交前至少自检以下内容：

1. 是否破坏了分层边界。
2. 是否新增了未纳入统一结果模型的输出字段。
3. 是否在用例层误写了技术定位细节。
4. 是否遗漏了敏感信息脱敏。
5. 是否补充了必要注释和对应测试。
6. 是否同步更新了文档和模板。

## 15. 默认执行约定

在没有额外说明的情况下，后续实现默认遵循以下执行约定：

1. 文档和方案说明优先中文。
2. 字段名、对象名、关键字、代码标识使用英文。
3. 所有输出产物以统一结果模型为中心生成。
4. 优先实现 `V1` 闭环，不提前扩展 `V2/V3` 功能。
